Linux驱动之MTD子系统（一）Block设备基础

发表于 2020-10-06 分类于 Linux Driver Disqus：
本文字数： 17k 阅读时长 ≈ 16 分钟

1. blk_mq介绍

1.1 Background

从内核开发之初，磁硬盘一直是事实上的标准。 Block IO 子系统旨在为那些在进行随机访问时具有高阻塞的设备实现最佳性能，而机械运动部件是瓶颈，比存储堆栈上的任何层慢得多。这种优化技术的一个例子涉及根据硬盘磁头的当前位置对读/写请求进行排序。

然而，随着固态驱动器和非易失性存储器的发展，没有机械部件，也没有随机访问阻塞，能够执行高并行访问，堆栈的瓶颈已经从存储设备转移到了操作系统。为了利用这些设备设计中的并行性，引入了多队列机制。

前一种设计有一个单独的队列来存储带有单个 Block IO 请求。由于缓存中的脏数据和多个处理器的单一锁的瓶颈，这在 SMP 系统中不能很好地扩展。当不同的进程（或同一进程，移动到不同的 CPU）想要执行Block IO 时，这种设置也会遇到拥塞。取而代之的是blk-mq API 会生成多个队列，这些队列具有 CPU 本地的各个入口点，从而无需锁定。

1.2 Operation

当用户空间对Block设备执行 IO（例如读取或写入文件）时，blk-mq 会采取行动：它将存储和管理对Block设备的 IO 请求。

blk-mq 有两组队列：软件暂存队列和硬件调度队列。当请求到达Block层时，它会尝试尽可能最短的路径：直接将其发送到硬件队列。但是，有两种情况它可能不会这样做：如果在该层附加了 IO 调度程序，或者如果我们想尝试合并请求。在这两种情况下，请求都会被发送到软件队列。

然后，在软件队列处理完请求后，它们将被放置在硬件队列中，第二阶段队列是硬件可以直接访问以处理这些请求。但是，如果硬件没有足够的资源来接受更多的请求，blk-mq 会将请求放置在一个临时队列中，以便将来在硬件有能力时发送。

1.3 Software staging queues

block IO 子系统将请求添加到软件暂存队列（由 struct blk_mq_ctx 表示）中，以防它们没有直接发送到驱动程序。一个请求是一个或多个 BIO。他们通过数据结构struct bio到达Block层。然后Block层会从中构建一个新的结构，这个结构请求将用于与设备驱动程序进行通信。每个队列都有自己的锁，队列的数量由每个 CPU 或每个node定义。

分段队列可用于合并相邻扇区的请求。例如，对扇区 3-6、6-7、7-9 的请求可以成为对 3-9 的一个请求。即使与顺序访问相比，对 SSD 和 NVM 的随机访问具有相同的响应时间，顺序访问的分组请求也会减少单个请求的数量。这种合并请求的技术称为Plugging。

除此之外，可以通过 IO 调度程序重新排序请求以确保系统资源的公平性（例如，确保没有应用程序遭受饥饿）和/或提高 IO 性能。

1.4 IO Schedulers

Block层实现了多个调度程序，每个调度程序都遵循启发式方法来提高 IO 性能。它们是“可插拔的”（如即插即用），因为它们可以在运行时使用 sysfs 进行选择。你可以在 https://www.kernel.org/doc/html/latest/block/index.html`_ 阅读更多关于 Linux 的 IO 调度器。调度只发生在同一个队列中的请求之间，所以不能合并来自不同队列的请求，否则会出现缓存垃圾，并且需要为每个队列都有一个锁。在调度之后，请求有资格被发送到硬件。要选择的可能调度程序之一是 NONE 调度程序，这是最直接的一种。它只会将请求放置在进程正在运行的任何软件队列上，而无需任何重新排序。当设备开始处理硬件队列中的请求（即运行硬件队列）时，映射到该硬件队列的软件队列将根据它们的映射顺序排空。

1.5 Hardware dispatch queues

硬件队列（由struct blk_mq_hw_ctx表示）是设备驱动用来映射设备提交队列（或设备DMA环形缓冲区）的结构，是底层设备驱动取得所有权之前的Block层提交代码的最后一步请求。为了运行这个队列，Block 层从相关的软件队列中删除请求并尝试分派到硬件。

如果无法将请求直接发送到硬件，它们将被添加到请求的链表 (hctx->dispatch) 中。然后，下一次Block 层运行队列时，它将首先发送位于“调度”列表中的请求，以确保对那些准备好首先发送的请求进行公平调度。硬件队列的数量取决于硬件及其设备驱动支持的硬件上下文的数量，但不会超过系统的核心数量。在这个阶段没有重新排序，每个软件队列都有一组硬件队列来发送请求。

Block层和设备协议都不能保证请求的完成顺序。 这必须由更高层处理，例如文件系统。

1.6 Tag-based completion

为了指示哪个请求已经完成，每个请求都由一个整数标识，范围从 0 到调度队列大小。 此标记由Block层生成，稍后由设备驱动程序重用，无需创建冗余标识符。 当一个请求在驱动程序中完成时，标签被发送回Block层以通知它完成。 这消除了进行线性搜索以找出已完成的 IO 的需要。

2.bio

2.1 Scope for tuning the generic logic to satisfy various requirements

Block 层设计支持自适应抽象来处理常见处理，并能够根据设备的性质和调用者的要求将逻辑调整到适当的程度。 重写的目标之一是提高可调性程度并使更高级别的代码能够最大程度地利用底层设备/驱动程序功能以获得更好的 I/O 性能。 鉴于不断改进的硬件功能和旨在利用这些功能的应用程序/中间件软件，这尤其重要。

2.1.1 Tuning based on low level device / driver capabilities

具有大型内置缓存、智能 I/O 调度优化、内存 DMA 支持等的复杂设备可能会发现一些通用处理开销，而对于功能较弱的设备，通用功能对于性能或正确性原因至关重要。应该在通用 Block 层使用设备的某些功能或参数的知识，以代表驱动程序做出正确的决定。

这是如何实现的？

在每个队列级别进行调优：

Per-queue limits/values exported to the generic layer by the driver

通用 i/o 调度程序逻辑使用的各种参数在每个队列级别设置（例如最大请求大小、分散-聚集列表中的最大段数、逻辑块大小）

以前作为由主要/次要索引的全局数组可用的一些参数现在直接与队列相关联。其中一些将来可能会进入块设备结构。一些特性已被合并到队列标志字段中，而不是单独的字段。有 blk_queue_xxx 函数来设置参数，而不是直接更新字段。

一些新的队列属性设置：

启用到 high-memory page的 I/O，dma_address 是限制。没有 high-memory 默认值。

设置两个限制请求大小的变量。

  • 请求队列的 max_sectors，这是一个以 512 字节扇区为单位的软大小，可以由核心内核动态改变。
  • 请求队列的 max_hw_sectors，这是一个硬限制，反映了驱动程序可以以 512 字节扇区为单位处理的最大请求大小。

max_sectors 和 max_hw_sectors 的默认值都是255. max_sectors 的上限为 1024。

您可以在请求中处理的最大物理段数。 128 默认

硬件可以在请求中处理的最大 dma 段。 128 默认值（主机适配器限制，在 dma 重新映射后）。

集群段的最大大小，默认为 64kB。

硬件可以操作的最小扇区大小，默认为 512 字节。

新队列标志：

  • QUEUE_FLAG_CLUSTER
  • QUEUE_FLAG_QUEUED
High-mem i/o capabilities are now considered the default

通用bounce 缓冲区逻辑，假设驱动程序无法直接处理它，Block 层默认将 high-memory 缓冲区上的 i/o 请求复制/输出到low-memory缓冲区在 2.5 中更改。弹跳逻辑现在仅适用于设备无法处理 i/o 的内存范围。驱动程序可以通过为设备的请求队列设置队列bounce 限制 (blk_queue_bounce_limit()) 来指定这一点。这避免了设备能够处理 high-memory i/o 时拷入/拷出的低效率。

为了在设备能够支持的情况下启用 high-memory i/o，现在已修改 pci dma 映射routine 和相关数据结构以完成直接 page -> bus 转换，而无需虚拟地址映射（不像较早的virtual address -> bus translation转换）。所以这对于 high-memory page（没有相应的内核虚拟地址空间映射）和low-memorypage都是一致的。

注意：请参阅 Documentation/core-api/dma-api-howto.rst 以讨论 PCI high-memory DMA 方面和分散收集列表的映射，以及对 64 位 PCI 的支持。

仅当 I/O 需要在超出设备支持的物理内存地址的page上发生的情况下才需要特殊处理。在这些情况下，代表受支持内存范围中的缓冲区的bounce bio 用于根据操作类型根据需要使用 copyin/copyout 执行 i/o。例如，在读取操作的情况下，读取的数据必须在 i/o 完成时复制到原始缓冲区，因此设置回调routine 来执行此操作，而对于写入操作，则在发出操作之前从原始缓冲区复制数据到bounce缓冲区。由于原始缓冲区可能位于未映射到内核虚拟地址中的 high-memory 区域，因此执行复制可能需要 kmap 操作，并且在完成路径中可能需要特别小心，因为它可能不在 irq 上下文中。分配bounce缓冲区时还需要特别小心（通过 GFP 标志），以避免某些 high-memory 死锁的可能性。

也有可能会从 high-memory 区域分配一个bounce buffer，该区域没有映射到内核虚拟地址，但在设备可以直接使用的范围内；因此在复制操作期间可能需要对跳出page进行映射。

在某些情况下，即使不需要 bounce 缓冲区，也可能需要对来自high-memory的page进行映射。例如，设备可能需要中止 DMA 操作并恢复到 PIO 以进行传输，在这种情况下，需要page的虚拟映射。对于 SCSI，在无法信任low level 驱动程序正确处理单个 sg 条目的某些情况下，也会执行此操作。driver应在适当的情况下根据需要执行 kmaps。如果需要，驱动程序还可以单独使用 blk_queue_bounce() routine 将 highmem i/o bounce到特定请求的 low-memory 。

The i/o scheduler algorithm itself can be replaced/set as appropriate

在前面讲到，可以为特定队列Plugging全新的 i/o 调度程序或从（复制）现有的通用调度程序中选择并替换/覆盖其中的某些部分。 3.1 重写提供了改进的 i/o 调度程序模块化。 有更多的可插拔回调，例如用于初始化、添加请求、提取请求，这使得在通用循环之外抽象特定的 I/O 调度算法方面和细节成为可能。 它还可以对块驱动程序完全隐藏 i/o 调度程序的实现细节。

将使用 I/O 调度程序包装器而不是直接访问队列。 有关详细信息，请参阅第 I/O 调度程序。

2.1.2 Tuning Based on High level code capabilities

Application capabilities for raw i/o

这来自一些高性能数据库/中间件要求，其中应用程序更喜欢根据对访问模式和 I/O 特性的理解做出自己的 I/O 调度决策

High performance filesystems or other higher level kernel code's capabilities

文件系统等内核组件也可以采用自己的 I/O 调度决策来优化性能。 日志文件系统可能需要对 I/O 排序进行一些控制。

通用Block 层对此有什么样的支持？

bio 结构中的标志和 rw 字段可用于从上面进行一些调整，例如指示 i/o 只是预读请求或优先级设置（当前未使用）。 就用户应用程序而言，他们需要一种额外的机制，通过开放标志或 ioctls，或其他一些上层机制来传达此类设置以阻止。

  • Request Priority/Latency

    Arjan 提出的请求优先级方案允许更高级别对 i/o 请求与队列中其他未决请求的优先级进行一些广泛的控制（高/中/低）。 例如，它允许读取按需引入可执行page的优先级高于未在队列中老化太多的待处理写入请求。 潜在地，这个优先级甚至可以以某种方式暴露给应用程序，提供更高级别的可调性。 基于时间的老化避免了较低优先级请求的饥饿。 bio 结构中 bi_opf 标志字段中的某些位旨在用于此优先级信息。

2.1.3 Direct Access to Low level Device/Driver Capabilities (Bypass mode)

在某些情况下，high-level code需要直接访问low level 设备功能，或者需要能够绕过某些中间 I/O 层向设备发出命令。例如，这些可能是通过 ioctl 接口发出的特殊控制命令，或者可能是强调驱动器在某些类型的健康测试中的能力的原始读/写命令。无需通过上层即可在多个级别拥有直接接口，这使得从媒体开始逐层执行自下而上的 I/O 路径验证成为可能。

对于此类 ioctl 或诊断接口通常会使用的特制请求，可以绕过正常的 I/O 提交接口，例如 submit_bio，并且可以使用电梯 add_request routine 直接将此类请求Plugging队列或最好是 blk_do_rq routine 可用于将请求放入队列并等待完成。或者，有时调用者可能只是使用请求作为参数调用较low level 别的驱动程序特定接口。

如果请求是传递与命令相关的特殊信息的一种方式，则此类信息与 request->special 字段相关联（而不是误用 request->buffer 字段，该字段用于请求数据缓冲区的虚拟映射）。

为了传递请求数据，如果底层驱动程序解释 bio 段或使用Block 层 endrequest 函数来完成 i/o，调用者必须建立一个 bio 描述符来表示相关的内存缓冲区。或者，可以直接使用 request->buffer 字段来指定缓冲区的虚拟地址，如果驱动程序期望以这种方式传递的缓冲区地址并忽略所涉及的请求类型的 bio 条目。在后一种情况下，驱动程序将修改和管理 request->buffer、request->sector 和 request->nr_sectors 或 request->current_nr_sectors 字段本身，而不是使用Block 层 end_request 或 end_that_request_first 完成接口。 （请参阅 2.3 或 Documentation/block/request.rst 以获取对请求结构字段的简要说明）

Pre-built Commands

可以使用预先构建的自定义命令创建请求以直接发送到设备。请求结构中的 cmd 块有填充命令字节的空间。 （即 rq->cmd 现在大小为 16 字节，用于命令预构建，请求的类型现在通过 rq->flags 而不是通过 rq->cmd 指示）。

在这种情况下，可以设置请求结构标志来指示请求的类型（REQ_PC：传递给驱动程序的直接数据包命令，REQ_BLOCK_PC：通过 blk_do_rq 发出的数据包命令，REQ_SPECIAL：特殊请求）。

它可以帮助提前为请求预先构建设备命令。驱动程序现在可以指定请求准备函数 (q->prep_rq_fn)，Block 层将调用该函数为给定请求预构建设备命令，或为请求执行其他准备处理。该routine 由 elv_next_request() 调用，即通常在服务请求之前调用。 （对于启用了 RQF_DONTPREP 的请求，不会调用准备函数）

预构建甚至可以提前完成，即在将请求放入队列之前，而不是在驱动程序中即时构建命令，同时为请求队列提供服务，因为它可能会影响中断上下文中的延迟或一般响应。 添加早期预构建的一种方法是在我们未能合并请求时执行此操作。 现在在请求标志中设置 REQ_NOMERGE 以在将来跳过此标志，这意味着在我们将其提供给设备之前它不会更改。 所以可以在那里调用 pre-builder 钩子。

2.2 Flexible and generic but minimalist i/o structure/descriptor

2.2.1 Reason for a new structure and requirements addressed

在 2.5 之前，buffer head被用作通用Block 层的 i/o 单元，low level 请求结构与连续 i/o 请求的buffer head链相关联。 当底层设备能够一次性处理 i/o 时,这导致在处理大型 i/o 请求和 readv/writev 样式操作时效率低下，因为它迫使此类请求在传递到通用Block 层之前被分解成小块，仅由 i /o 调度程序。 此外，将buffer head用作并非源自缓冲区缓存的 i/os 的 i/o 结构，不必要地增加了为每个此类块生成的描述符的权重。

以下是在 2.5 中重新设计块 i/o 数据结构时考虑的一些目标和期望。

  1. 应该适合作为 raw 和 buffered i/o 的描述符 - 避免与直接/page i/o 路径无关的缓存相关字段，或可能与raw i/o 无关的文件系统块大小对齐限制.
  2. 能够表示high-memory buffers（在内核地址空间中没有虚拟地址映射）。
  3. 能够表示大型 i/os，而无需不必要地将它们分解（即在一次拍摄中大于 PAGE_SIZE 块）
  4. 同时，能够保留来自不同来源的 i/o 或需要单独完成的 i/o 单元的独立身份（例如出于延迟原因）
  5. 能够表示涉及多个物理内存段（包括非page对齐的page片段，如通过 readv/writev 指定的）的 i/o，而不会不必要地将其分解，前提是底层设备能够处理它。
  6. 最好应该基于一个内存描述符结构，该结构可以在不同类型的子系统或层之间传递，甚至可能是网络，在此过程中没有重复或额外的数据/描述符字段副本
  7. 当结构通过分层驱动程序（lvm、md、evms）时，能够以最小的开销处理拆分/合并的可能性。

解决方案是为Block 层定义一个新的结构（bio），而不是直接使用buffer head结构（bh），其想法是避免一些相关的包袱和限制。 bio结构统一用于block层的所有i/o； 它构成了缓冲 i/o 的 bh 结构的一部分，在raw/direct i/o 的情况下，kiobuf 被映射到 bio 结构。

2.2.2 The bio struct

bio 结构使用指向 <page, offset, len> 元组数组的向量表示来描述 i/o 缓冲区，并具有描述 i/o 参数和状态的各种其他字段，这些字段需要维护以执行 i /o。

请注意，这种表示意味着 bio 根本没有虚拟地址映射（与buffer head不同）。

  • 可以使用由 <page, offset, len> 片段数组组成的 bio_vec 列表一次性发送大型 i/os（类似于片段在零拷贝网络代码中的表示方式）。
  • 跨多个设备拆分 i/o 请求（如 lvm 或 raid 的情况）是通过克隆 bio（其中克隆指向相同的 bi_io_vec 数组，但索引和大小相应修改）来实现的。
  • 一个 bios 链表像以前一样用于不相关的合并 [#]_ - 这避免了重新分配并使独立完成更容易处理。
  • 遍历 req 列表的代码可以使用 rq_for_each_segment 找到 bio 的所有段。这处理了一个请求有多个 bios 的事实，每个 bios 可以有多个段。
  • 无法一次性处理大型 bio 的驱动程序可以使用 bi_ite 字段来迭代要处理的下一个 bio_vec 条目。 （例如，对于 IDE，1MB bio_vec 需要以最大 128kB 的块处理）[TBD：最好还应该有 bi_voffset 和 bi_vlen 以避免修改 bi_offset 和 len 字段]

bi_end_io() i/o 回调在整个 bio 的 i/o 完成时被调用。

在lower level，驱动程序从合并的 bios 构建一个分散收集列表。分散收集列表采用 <page, offset, len> 条目数组的形式，并在适当的时候填充了它们相应的 dma 地址映射。作为一种优化，连续的物理page可以被单个条目覆盖，其中 指的是第一页，而 覆盖page范围（这种方式最多可以覆盖 16 个连续page）。有一个帮助程序 (blk_rq_map_sg)，驱动程序可以使用它来构建 sg 列表。

注意：现在只有一个page的 bios 用户是 ll_rw_kio，这意味着只有原始 I/O 使用它（直接 I/O 现在可能无法工作）。然而，目的是使page等的集群成为可能。 SGI 的 pagebuf 抽象层也使用multi-page bios，但目前未包含在库存开发内核中。 Andrew Morton 正在进行的multi-page生物写出和预读补丁也是如此。

request 结构是传递给 low level 驱动程序的结构。 Block 层 make_request 函数建立一个request结构，将其放入队列并调用驱动程序 request_fn。 驱动程序使用Block 层辅助routine elv_next_request 将下一个请求从队列中拉出。 控制或诊断功能可能会绕过Block 并直接调用传入特殊构造的 request 结构的 low level 驱动程序入口点。

在buffer驻留在 low-memory 中的情况下，缓冲区字段只是 I/O 缓冲区当前段的虚拟地址映射。对于 high memory i/o，此字段无效，驱动程序不得使用。

2.3 Using bios

2.3.1 Setup/Teardown

有用于管理分配、引用计数和释放 bios（bio_alloc、bio_get、bio_put）的 routines。

这利用了 Ingo Molnar 的内存池实现，它使 bio 等子系统能够维护自己的预留内存池，以保证在极端 VM 负载期间无死锁分配。例如，VM 子系统利用 Block 层来写出脏页以释放内存空间，这种情况需要谨慎处理。在无法通过正常方式进行分配的情况下，分配逻辑从预先分配的应急储备中提取。如果池是空的并且可以等待，那么它将触发有助于释放内存或补充池（没有死锁）并等待池中的可用性的操作。如果它在 IRQ 上下文中，则无法执行此操作，则在池为空时分配可能会失败。一般来说，mempool 总是首先尝试执行分配而不必等待，即使这意味着挖掘池，只要它不低于内存的 50% 。

在空闲时，内存被释放到池中或直接释放，这取决于池中当前的可用性。 mempool 接口让子系统指定用于正常分配和释放的 routines。在 bio 的情况下，这些routines使用标准的 slab 分配器。

bio_alloc 的调用者应该采取某些步骤来避免死锁，例如避免在已经持有从池中获得的内存时尝试从池中分配更多内存。

从池中分配的内存应该在有限的时间内释放回来（在 bio 的情况下，这将是在 i/o 完成之后）。这确保如果池的一部分已用完，则某些工作（在本例中为 i/o）必须已经在进行中，并且当它结束时内存将可用。如果从同一代码路径中的多个池进行分配，则分配的顺序或层次结构需要一致，就像处理多个锁一样。

bio_alloc routines 还需要为 non-clone bio 分配 bio_vec_list (bvec_alloc())。为不同大小的 biovec 设置了 6 个池，因此 bio_alloc(gfp_mask, nr_iovecs) 将从这些块中分配给定大小的 vec_list。

bio_get() routines 可用于在 i/o 提交之前保存对 bio 的额外引用，如果 bio 字段可能在 i/o 发出后被访问（因为 bio 可能会在 i/o 发布后被释放） /o 完成同时发生）。

该bio_clone_fast() routines 可用于复制 bio，其中克隆与原始 bio 共享 bio_vec_list（即，两者都指向相同的 bio_vec_list）。这通常用于在 lvm 或 md 中拆分 i/o 请求。

2.3.2 Generic bio helper Routines

Traversing segments and completion units in a request

宏 rq_for_each_segment() 应该用于遍历请求列表中的 bios（驱动程序应避免直接尝试自己做）。使用这些helpers还应该可以更轻松地应对将来的块更改。

I/O 完成回调是按per-bio而不是按per-segment进行的，因此在完成时遍历per-bio链的驱动程序需要牢记这一点。不区分段和完成单元的驱动程序需要重新组织以支持 multi-segment bios。

Setting up DMA scatterlists

2.4 The I/O scheduler

I/O 调度器，又名电梯，在通用调度队列和特定 I/O 调度程序两层中实现。除非另有说明，电梯用于将部件和 I/O 调度器都指代特定的 I/O 调度器。

块层在block/*.c 中实现通用调度队列。通用调度队列负责重新排队、处理非 fs 请求和所有其他微妙之处。

特定的 I/O 调度器负责对正常的文件系统请求进行排序。他们还可以选择延迟某些请求以提高吞吐量或出于任何目的。如复数形式所示，有多个 I/O 调度程序。它们可以构建为模块，但至少应该在内核中构建一个。每个队列可以选择不同的队列，也可以动态更改为另一个队列。

对 i/o 调度程序的块层调用遵循约定 elv_xxx()。这会在电梯开关 (block/elevator.c) 中调用电梯_xxx_fn。哦，xxx 和 xxx 可能不完全匹配，但请发挥您的想象力。如果电梯没有实现某种功能，开关就不会做任何事情或做一些最小的家务工作。

2.4.3 I/O scheduler implementation

通用 i/o 调度程序算法尝试排序/合并/批处理请求以获得最佳磁盘扫描和请求服务性能（基于通用原则和设备功能），优化用于：

1、提高吞吐量 2、改善延迟 3、更好地利用硬件和 CPU 时间

Characteristics:

  • 二叉树
    AS和deadline i/o调度器使用红黑二叉树进行磁盘位置排序和搜索，使用fifo链表进行基于时间的搜索。这提供了良好的可扩展性和良好的信息可用性。请求几乎总是按磁盘排序顺序发送的，因此按排序顺序保存下一个请求的缓存以防止二叉树查找。 然而，这种安排不是通用的块层特性，因此电梯可以随意实现队列。

  • Merge hash
    AS 和截止日期使用由请求的最后一个扇区索引的哈希表。这使得合并代码能够快速查找“反向合并”候选对象，即使在一个磁盘上同时执行多个 I/O 流时也是如此。 由于大多数 I/O 模式的性质，“前合并”，即在现有请求前面合并的新请求，远不如“后合并”常见。前合并由 AS 中的二叉树和截止日期调度程序处理。

  • Plugging the queue to batch requests in anticipation of opportunities for merge/sort optimizations
    Plugging是当前 i/o 调度算法采用的一种方法，以便它在队列中收集足够的请求，以便能够利用电梯中的排序/合并逻辑。如果请求进来时队列是空的，那么它会Plugging请求队列（有点像塞住容器的浴缸以积聚液体）直到它填满更多请求，然后才开始为请求提供服务要求。这提供了在将请求传递给设备之前合并/排序请求的机会。当队列被unplugging（以再次打开流）时，有多种情况，要么是通过计划任务，要么是按需。例如wait_on_buffer 设置通过sync_buffer() 运行blk_run_address_space(mapping) 的unplugging。或者调用者可以通过 blk_unplug(bdev) 明确地做到这一点。

Aside:
这是一个有争议的领域，因为目前尚不清楚堵塞是否总是正确的做法。设备通常有自己的队列，允许在软件中建立一个大队列，而让设备空闲一段时间可能并不总是有意义的。诀窍是处理何时插入和何时打开之间的良好平衡。此外，现在我们有一个多页 bios 正在排队，我们可能不需要等待合并来自碎片的大请求。

2.5 Scalability related changes

2.5.1 Granular Locking: io_request_lock replaced by a per-queue lock

从 2.5 开始，全局 io_request_lock 已被删除，以避免它引起的可扩展性瓶颈，并已被更细粒度的锁定所取代。request队列结构有一个指向要用于该队列的锁的指针。因此，现在可以按队列进行锁定，并在必要时提供跨队列共享锁的规定（例如，scsi 层将队列锁指针设置为相应的适配器锁，这会导致每个主机的锁定粒度）。锁定语义是相同的，即锁定仍然由块层施加，在 request_fn 执行之前获取锁定，这意味着许多旧驱动程序仍然应该是 SMP 安全的。如果需要，驱动程序可以自行删除队列锁。需要相应修改显式使用 io_request_lock 进行序列化的驱动程序。

并将该锁的地址传递给 blk_init_queue()。

2.5.2 64 bit sector numbers (sector_t prepares for 64 bit support)

bio 结构中使用的扇区号已更改为sector_t，可以将其定义为 64 位以准备支持 64 位扇区。

4. write back cache control

4.1 Introduction

许多存储设备，尤其是在消费市场中，都带有易失性回写缓存。这意味着设备在数据实际到达非易失性存储器之前向操作系统发出 I/O 完成信号。这种行为显然会加快各种工作负载的速度，但这意味着操作系统在执行数据完整性操作（如 fsync、sync 或 unmount）时需要将数据强制输出到非易失性存储。

Linux 块层提供了两种简单的机制，让文件系统控制存储设备的缓存行为。这些机制是强制缓存刷新和请求的强制单元访问 (FUA) 标志。

4.2 Explicit cache flushes

REQ_PREFLUSH 标志可以被 OR ed 到从文件系统提交的 bio 的 r/w 标志中，并确保在实际 I/O 操作开始之前已刷新存储设备的易失性缓存。这明确地保证了先前完成的写入请求在标记 bio 开始之前位于非易失性存储上。此外，REQ_PREFLUSH 标志可以设置在一个空的 bio 结构上，这只会导致没有任何依赖 I/O 的显式缓存刷新。建议使用blkdev_issue_flush()helper 进行纯缓存刷新。

4.3 Forced Unit Access

REQ_FUA 标志可以 OR ed 到从文件系统提交的 bio 的 r/w 标志中，并确保仅在数据提交到非易失性存储后才发出此请求的 I/O 完成信号。

4.4 Implementation details for filesystems

文件系统可以简单地设置 REQ_PREFLUSH 和 REQ_FUA 位，而不必担心底层设备是否需要任何显式缓存刷新以及如何实现强制单元访问。REQ_PREFLUSH 和 REQ_FUA 标志都可以在单个 bio 上设置。

4.5 Implementation details for bio based block drivers

这些驱动程序将始终看到 REQ_PREFLUSH 和 REQ_FUA 位，因为它们位于 submit_bio 接口正下方。对于重新映射驱动程序，需要将 REQ_FUA 位传递到底层设备，并且需要为具有 REQ_PREFLUSH 位设置的 bios 实现全局刷新。对于没有易失性缓存的真实设备驱动程序，可以简单地忽略非空 bios 上的 REQ_PREFLUSH 和 REQ_FUA 位，无需任何工作即可成功完成没有数据的 REQ_PREFLUSH 请求。具有易失性缓存的设备的驱动程序需要自己实现对这些标志的支持，而无需块层的任何帮助。

4.6 Implementation details for request_fn based block drivers
