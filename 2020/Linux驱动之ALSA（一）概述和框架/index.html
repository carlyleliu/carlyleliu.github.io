Linux驱动之ALSA（一）概述和框架

发表于 2020-10-16 分类于 Linux Driver Disqus：
本文字数： 3.3k 阅读时长 ≈ 3 分钟

1. 概述

高级 Linux 声音架构 (ALSA) 为 Linux 操作系统提供音频和 MIDI 功能。ALSA 具有以下显着特点：

  • 有效支持所有类型的音频接口，从消费级声卡到专业的多声道音频接口。
  • 完全模块化的声音驱动器。
  • SMP 和线程安全设计。
  • 用户空间库 (alsa-lib) 用于简化应用程序编程并提供更高级别的功能。
  • 支持较旧的 Open Sound System (OSS) API，为大多数 OSS 程序提供二进制兼容性。

2. 一些重要概念

2.1 I2S

I2S全称Inter-IC Sond Bus，是飞利浦在1986年定义(1996年修订)的数字音频传输标准，用于数字音频数据在系统内部器件之间传输，例如编解码器Codec、DSP、数字输入/输出接口、ADC、DAC和数字滤波器等。 对于I2S的数字接口定义也比较简单，没有从地址或者从设备的概念，在I2S总线上，只能同时存在一个主设备和发送设备。在I2S系统中，提供时钟（SCK）的设备为主设备，其常见的系统框图如下：

在I2S传输协议中，数据信号、时钟信号以及控制信号是分开传输的。I2S协议只定义三根信号线：时钟信号SCK、数据信号SD和左右声道选择信号WS。

  • SCLK
    时钟信号:模块内的同步信号，主模式时由模块内部自己产生，从模式由外部提供.

  • SD
    数据信号:在WS变化后的第一个SCK脉冲，先传输最高位（MSB, Most Significant Bit）。先传送MSB是因为发送设备和接收设备的字长可能不同.

  • WS
    左右声道选择信号 Word Select:WS＝0，表示选择左声道;WS＝1，表示选择右声道.WS也称帧时钟，即LRCLK/Left Right Clock。WS频率等于声音的采样率.

2.2 PCM/TDM

PCM = Pulse Code Modulation 是通过等时间隔（即采样率时钟周期）采样将模拟信号数字化的方法。下图是4bit采样速率的PCM数据量化示意图:

PCM接口常用于板级音频数字信号的传输，与I2S类似，其实I2S也是PCM的一种特例接口，只不过，I2S的速率会更高，比较适用于传音乐。而PCM通常用于AP处理器与通信MODEM之间的语言数据传输（就是双向打电话数据），对于I2S只能传2个声道的数据，而PCM可以传多达16路数据，采用时分复用的方式，也就是TDM。其接口与I2S类似，电路信号为:

  • PCM_CLK 数据时钟信号
  • PCM_SYNC 帧同步时钟信号
  • PCM_IN 接收数据信号
  • PCM_OUT 发送数据信号

2.3 PDM

PDM(Pulse Density Modulation)是一种数字信号表示模拟信号的调制方法，声音通过传感器获得模拟信号，经过AD，得到音频数字信号，然后经过PDM脉冲转换成数字信号。PDM使用远高于PCM采样率的时钟采用调制模拟分量，只有1位输出，要么是0，要么是1。

PDM方式的逻辑相对复杂，但只需要两根线，时钟和数据。对于下图，主设备为两个从设备提供时钟，分别在时钟的上升沿和下降沿触发选择Source 1/2作为数据输入。

PDM在诸如手机和平板等对于空间限制严格的场合有着广泛的应用前景。在数字麦克风领域，应用最广的就是PDM接口，其次为I2S接口。

2.4 S/PDIF

S/PDIF的全称是Sony/Philips Digital Interface Format，由于被广泛采用，它成为事实上的民用数字音频格式标准，大量的消费类音频数字产品如民用CD机、DAT、MD机、计算机声卡数字口等都支持S/PDIF，在不少专业设备上也有该标准的接口。就传输方式而言，SPDIF分为输出（SPDIF OUT）和输入（SPDIF IN）两种。

3. 驱动框架

ALSA驱动程序框架如下：原图

  • ALSA-Lib
    APP应用程序只要调用 alsa-lib 提供的 API接口来实现放音、录音、控制。现在提供了两套基本的库，tinyalsa是一个简化的alsa-lib库，现在Android的系统中主要使用它。

  • ALSA-Core
    alsa 核心层，向上提供逻辑设备（PCM/CTL/MIDI/TIMER/…）系统调用，向下驱动硬件设备。

  • ASOC
    是 ALSA 的标准框架，是 ALSA-driver 的核心部分，提供了各种音频设备驱动的通用方法和数据结构，为 Audio driver提供 ALSA Driver API。

  • Hardware-Driver
    频硬件设备驱动，由三大部分组成，分别是 Machine、Platform、Codec，提供的 ALSA Driver API 和相应音频设备的初始化及工作流程，实现具体的功能组件，这也是驱动开发人员需要具体实现的部分。

4. ALSA数据结构

整体数据结构如下：原图

  • snd_minors[SNDRV_OS_MINORS]是一个全局数组里面存放的是系统所有的snd_minor结构体，一个snd_minor结构体里包含一个snd_card结构体。
  • snd_card结构体就是声卡设备的抽象。一个系统中的一个声卡设备都会通过这个数据结构进行描述，这是一个顶层数据结构。
  • snd_device就是具体声卡设备实例的抽象，可以是pcm设备、control设备、timer设备。系统通过snd_device结构体将一个声卡需要的音频流、控制定时器等设备连接在一起共同协同工作。而系统中存在多个声卡设备则存在多个snd_card结构体。
  • snd_device_ops是snd_device的操作函数集，其实就是实现设备的注册和释放等工作。

至于下面其他数据结构会在后续的文章中再进行描述，该图后面也还会再出现，没有将这些数据结构分开是为了可以在整体上对ALSA的实现可以有一个整体认识。这正是梳理数据结构之间关系的意义所在。
